# MIMIC-IV 患者模拟数据集 · 数据处理方案使用手册

> 配套论文: *Patients Are Not Oracles: Evaluating Interactive Clinical Reasoning beyond Complete-Information QA*
>
> 本手册说明 `data/mimic-iv/data_processing/` 下脚本的设计、用法与产物格式。目标是把静态的 MIMIC-IV 电子病历重建成**按时间分层、可门控**的病例 JSON，作为 Doctor / Patient / Exam 三智能体问诊流程的"底料"。

---

## 1. 这套脚本在做什么

MIMIC-IV 本身**没有对话**，只有回顾性 EHR。本方案的核心是把它转化为：

- **患者剧本（Patient agent 输入）**：入院时患者已知的病史；
- **检查环境（Exam agent 输入）**：医生下单后才返回的客观检查，**出院检查除外**；
- **隐藏答案（Evaluator 用）**：最终诊断，对两个 agent 都不可见。

一句话：**把"整段住院叙事"按时间切成三个平面，杜绝 Patient agent "未卜先知"。**

```
                出院小结 + ED 分诊 + labs/radiology/micro
                                │
        ┌───────────────────────┼───────────────────────┐
        ▼                       ▼                        ▼
 【入院时·患者可知】       【客观·检查可得】           【目标·隐藏】
     history(G_hist)        exams(G_exam)            diagnosis_hidden(G_dx)
  主诉/现病史/既往史/      入院查体/诊断期化验/        主诊断 ICD(seq_num=1)
  社会史/家族史/过敏/      影像/微生物                 + 出院诊断文本
  入院用药                 (出院检查已剔除)            (Brief Hospital Course 绝不暴露)
        │                       │                        │
        ▼                       ▼                        ▼
  Patient Agent            Exam Agent               Evaluator(隐藏)
```

## 5. 管线分阶段（stages）

`build_dataset.py` 把流程拆成 6 个 stage，中间表持久化在 `output/mimic_work.duckdb`，便于**只重跑某一步**：

| stage | 作用 | 读取的大表 | 典型耗时* |
|---|---|---|---|
| `cohort` | 队列筛选 + 一病人一例 + 出院小结文本 | discharge(1.1G) | ~30s |
| `triage` | ED 分诊主诉/生命体征 + ED 诊断 | 小表 | ~2s |
| `labs` | 诊断期化验(半连接 + 时间窗) | labevents(2.5G) | ~3min |
| `radiology` | 诊断期影像报告 | radiology(780M) | ~15s |
| `micro` | 诊断期微生物 | microbiologyevents(117M) | ~10s |
| `assemble` | 分段 + 平面归位 + 写 JSON/JSONL | — | <1s |

\* 外置 USB 盘实测，仅供参考。

只重跑分段逻辑（改了 `note_sections.py` 后最常用，无需重扫大表）：

```bash
../../../.venv/bin/python build_dataset.py --stages assemble
```

只重建化验：`--stages labs,assemble`。

## 6. 队列纳入/排除标准（cohort）

在 `build_dataset.py: build_cohort()` 中实现：

- **纳入**：成人（入院年龄 ≥ 18）；**经 ED 入院**（与 `edstays` 可连）；有主诊断（`diagnoses_icd.seq_num = 1`）；有出院小结且长度 > `--min-note-len`（默认 1500）；住院时长 ≥ `--min-los-hours`（默认 24h）。
- **排除**：无需诊断推理的主诊断 —— ICD-10 的 `S/T/O/Z` 开头（外伤/产科/Z码），ICD-9 的 `E/V` 码与 800–999（外伤/中毒/补充码）。
- **排除院内死亡**（默认开）：`hospital_expire_flag = 1` 的病例剔除（死亡病例多伴持续恶化/无意识，与"可对话"冲突）。`--keep-expired` 可保留。
- **排除"无法对话"主诊断类别**：ICD-10 前缀命中 `--exclude-icd-prefixes`（默认 `R40.2`昏迷 / `I46`心脏骤停 / `G93.1`缺氧脑病）的剔除。
- **一病人一例（subject-disjoint）**：每个 `subject_id` 只取**出院小结最长**的一次住院，自动消除同一病人多次入院的相关性，且天然满足 train/test 不交叉。
- **组装期二次过滤**：
  - HPI 段长度 < `--min-hpi-len`（默认 200）的病例跳过（病史不足无法 simulate）。
  - **可对话性闸门**（默认开，`--keep-nonconversable` 关闭）：`note_sections.is_conversable()` 命中主诉黑名单（AMS / unresponsive / cardiac arrest / coma / found down 等）或 HPI 硬信号（intubated&sedated / unable to provide history / GCS<8 / obtunded 等）的病例剔除。**晕厥不在黑名单内，不会误杀**。
  - **病史来源标注**：`detect_history_source()` 给每例打 `history_source ∈ {patient, mixed, collateral}`（旁人/病历代述检测），写入 provenance；句子级精细判定交由 Stage B B1。

## 7. "出院检查除外" 是怎么落地的

这是本方案的关键红线，由**两道机制**共同保证：

1. **文本侧（入院查体）**：`note_sections.admission_physical_exam()` 在 Physical Exam 段中识别出院查体标记（`DISCHARGE EXAM` / `physical exam on/upon discharge` / `--DISCHARGE EXAM--` / `VSS on discharge` 等），在最早命中处截断，**只保留入院查体**。
2. **结构化侧（化验/影像/微生物）**：只取**诊断期时间窗**内的记录 —— 从 `COALESCE(edregtime, ed_intime, admittime)` 起算的 `--lab-window-hours`（默认 72h），且不晚于 `dischtime - --discharge-buffer-hours`。长住院靠 72h 窗口天然排除出院期检查；短住院则保留完整工作期。
3. **Markdown 侧**：`export_markdown.py` 还会从 Pertinent Results 原文中剔除 `LABS ON DISCHARGE` 之类的出院化验小节。

> ⚠️ 关于 `--discharge-buffer-hours`：默认 **0**。对 ~24h 的短住院设过大缓冲会把全部检查误删，请谨慎调大。

### 7.1 exam 时间窗上界 = 固定时间窗，**不是**"首次确定性治疗时间"

当前 exam 上界是 `LEAST(start_ts + lab-window-hours, dischtime − discharge-buffer-hours)`，即从入院/ED 登记起算的**固定 72h 诊断期窗口**，与"首次治疗/手术时刻"无关。

- 为什么暂不做"治疗时间硬上界"：① `--first-per-item` 已取每项最早一次，压掉了治疗后复查趋势泄漏；② 真正危险的是**确诊性检查**（手术病理 / HIDA / 确诊导管造影，往往*就是答案*），这类应在 Stage C 当作**可下单 exam 节点 + D_crit 候选**用 reveal policy 控制，而非靠时间窗一刀切；③ MIMIC-IV 里精确治疗时刻需拼 `poe/emar/prescriptions.starttime` 或 ICU `procedureevents`，噪声大、覆盖不全。
- 推荐做法：默认 72h，48h 作敏感性分析；exam 节点打 `latency` 标签；治疗时间硬上界列为 **v2 可选项**。详见 `plan/stage_cd_plan.md` §2/§5。

### 7.2 化验清洗（Stage A 确定性，A2/A3/A5）

- **A2 de-id 数值回填**：当结构化 `value` 被去标识成 `___` 或缺失、但 `valuenum` 有数时，用 `valuenum` 回填生成 `value_display`，杜绝"明明有数却显示 `___`"。Markdown 渲染优先读 `value_display`。
- **A3 垃圾行过滤**：剔除溶血/黄疸/脂血质量指数（itemid `50934/50947/51678`，即 H/I/L），以及无 `valuenum` 且 value 为血气标本描述/注释（`VEN./ART./INTUBATED./HOLD…`）的行；保留 PT/pH/C3/C4 等合法短标签与 NEG/POS/Trace 等质性结果。
- **A5 锚点落 provenance**：每例记录 `lab_window_start / lab_window_end / lab_anchor_time`（窗口内最早一次化验时刻），保证选点可复现。

## 8. 输出 JSON 结构（每例）

```jsonc
{
  "case_id": "inp_<subject_id>_<hadm_id>",
  "source": { "subject_id", "hadm_id", "stay_id", "note_id", "dataset" },

  "demographics": { "age", "gender", "race", "marital_status", "language" },

  // Doctor agent 初始唯一可见(分诊台呈现)
  "presentation": {
    "chief_complaint", "admission_type", "arrival_transport", "acuity",
    "triage_vitals": { "temperature","heartrate","resprate","o2sat","sbp","dbp","pain" }
  },

  // G_hist: 患者入院时可知 —— Patient agent 的剧本
  "history": { "chief_complaint","hpi","ros","pmh","psh",
               "social_history","family_history","allergies","medications_on_admission" },

  // G_exam: 检查可得(出院检查除外) —— Exam agent 的环境
  "exams": {
    "physical_exam_admission",     // 仅入院查体
    "pertinent_results_text",      // 出院小结里的结果摘要(文本)
    "labs":        [ { "itemid","label","fluid","category","charttime",
                       "value","valuenum","value_display",   // value_display: A2 回填后的可读值
                       "valueuom","ref_range_lower","ref_range_upper","flag" } ],
    "radiology":   [ { "note_type","charttime","text" } ],
    "microbiology":[ { "charttime","spec_type_desc","test_name",
                       "org_name","ab_name","interpretation" } ]
  },

  // G_dx: 隐藏答案 —— 两个 agent 都不可见, 仅 Evaluator 使用
  "diagnosis_hidden": {
    "primary_icd","icd_version","title","discharge_diagnosis_text",
    "ed_diagnosis": [ { "seq_num","icd_code","icd_title" } ]
  },

  "provenance": {
    "admittime","dischtime","los_hours",
    "icd_version_filter","lab_window_hours","discharge_buffer_hours",
    "lab_window_start","lab_window_end","lab_anchor_time",   // A5: 诊断期窗口与化验锚点
    "history_source","conversable","conversable_reasons",    // 病史来源 + 可对话性
    "note_len","note_deid_density","sections_found","n_labs","n_radiology","n_micro"
  }
}
```

> 化验默认每项只保留窗口内**最早一次**（`--first-per-item`，工作期快照）。需要全部时间点用 `--all-lab-values`。
>
> Stage B 通过的病例（`dataset.curated.jsonl`）会额外带一个 `qc` 块：`b1_source`（病史来源占比 + 句子标签）、`b2_admission`（可推理性/可解性打分）、`admitted`、`flag_for_review`。

## 9. Markdown 渲染（plain 版）

`export_markdown.py` 把每个 case JSON 渲染成临床风格 `.md`：基本信息 → 主诉分诊 → 病史 → 入院查体 → 化验表（异常值加粗 `⚠`）→ 影像 → 微生物 → 隐藏诊断（默认折叠在末尾）。支持**中英双语**。

```bash
../../../.venv/bin/python export_markdown.py                # 中文 → output/markdown/
../../../.venv/bin/python export_markdown.py --lang en      # 英文 → output/markdown/en/
../../../.venv/bin/python export_markdown.py --combined     # 额外合并成 all_cases.md
../../../.venv/bin/python export_markdown.py --hide-answer  # 纯问诊素材, 不含答案
```

| `--out-dir` | `./output/iaa` | 输出目录 |
| `--sample-clean` | 20 | 额外抽取的"干净通过"病例数（平衡 κ 估计） |
| `--max-cases` | 0 | 队列上限（0=不限） |
| `--seed` | 42 | 抽样随机种子 |

> 复核队列 = 全部被拒 + 全部带 `flag_for_review` + 随机干净样本。产出病例级 `iaa_cases.csv`（机器分数/标记 + 留空人工列）与句子级 `iaa_b1_sentences.csv`（B1 来源 IAA），两名标注者各填一份。

### score_iaa.py（一致性计算）

读两名标注者填好的 CSV，算 **Cohen's κ**（病例级 conversable/history/solvable/gold/decision + 序数 reasoning 用二次加权 κ；句子级 source 5 类），并报**人机一致性**（标注者 vs `machine_decision`/`b2_solvability`/`machine_source`，直接回应"LLM judge 准不准"）。零依赖，结果写 `output/iaa/iaa_agreement.md`。

```bash
../../../.venv/bin/python score_iaa.py \
    --cases-a annoA_cases.csv --cases-b annoB_cases.csv \
    --sent-a annoA_sent.csv  --sent-b annoB_sent.csv
# 只填了一名标注者时, 仅输出人机一致性:
../../../.venv/bin/python score_iaa.py --cases-a annoA_cases.csv
```

## 11. Stage B —— 质量闸门（B1/B2/B3）

`stage_b_gates.py` 读 `dataset.jsonl`，对每例打两组分（**只打分/打标签，绝不改写原文**），按阈值产出 `dataset.curated.jsonl`：

- **B1 Source Attribution**：HPI 逐句判定来源 ∈ {patient, collateral, chart_review, clinician_observed, osh_records}，给 `source_patient_ratio` —— 守住"患者本人能应答"的前提（G_hist 理应只采纳 patient 来源，精筛在 Stage C）。
- **B2 Case Admission**：输出 `diagnostic_reasoning_score / cc_dx_distance / gold_consistency / solvability / reasons` —— 拦掉"主诉≈答案"的 trivial 病例（GVI≈0，稀释指标）与证据不足无法推理的病例。
- **B3 Leakage Audit（确定性，无外部依赖）**：扫描暴露平面是否泄漏隐藏答案/事后信息 ——
  - `history_dx_leak`：gold 诊断词出现在患者**病史**(HPI/ROS) → 真泄漏（主诉里的同名症状已豁免）；
  - `exam_dx_mention`：gold 词出现在**检查**自由文本 → 信息项（提示哪个检查可能是 D_crit），非泄漏；
  - `tx_leak`：术后/转归词（post-op / …ectomy / path showed / was treated with 等）出现在暴露平面 → 事后信息泄漏。
  默认仅打 `dx_leak_in_history`/`treatment_leak` 标记转人工；`--reject-on-leakage` 时对病史泄漏硬拒绝。

两个后端：
- `--backend heuristic`（默认）：纯规则，**无任何外部调用**，本机即可跑，用作基线/冒烟。
- `--backend local`：调用项目根 `.env`（`API_KEY/MODEL/BASE_URL`）配置的 OpenAI 兼容端点，`temperature=0` + 受约束 JSON。**提示词与 0–5 评分标准集中在 `prompt.py`**（B1 五类来源定义；B2 给出每个分数的 rubric）。
  - ⚠ **DUA 合规闸**：脚本检测到 `BASE_URL` 为**非本地端点**时默认**中止**（凭证版 MIMIC 文本严禁外发）；仅在确认数据为 demo 开放数据时加 `--allow-external`。生产环境应把 `BASE_URL` 指向**本地模型端点**（vLLM/ollama 等）。

B2 LLM 输出额外含 `recommendation ∈ {admit, review, reject}`。产出：`stage_b_scores.jsonl`（每例打分 + 拒绝原因 + `flag_for_review`）、`dataset.curated.jsonl`（通过病例 + `qc` 块）、`stage_b_summary.json`（汇总）。`gold_consistency` 低 / 病史含旁人代述的病例**标记转人工**而非自动剔除。

## 13. 性能与排错

- **labevents 扫描慢**：2.5GB 全表扫描一次约数分钟（外置盘更慢）。调通后再 `--limit 0` 跑全量；只改分段逻辑时用 `--stages assemble` 免重扫。
- **`Binder Error: Ambiguous column`**：多见于 join 后未限定列名，已在脚本内处理。
- **去标识占位符 `___`**：`provenance.note_deid_density` 记录密度；密度过高的病例可在下游按需过滤。
- **类型错误**：所有 CSV 以 `all_varchar` 读入再显式 `TRY_CAST`，避免 MIMIC 脏数据触发推断失败。
